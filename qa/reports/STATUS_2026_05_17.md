# ⚠️ Schema/plan delta — Phase 0 plan needs a rewrite before §C–§J

The Phase 0 plan was authored against assumptions that do not hold in this repo.
Decision (this session): **pause and rewrite the plan against the real schema**
before implementing §C–§J. This section is the input for that rewrite.

## 1. Identity & auth model

| Plan assumes | Reality |
|---|---|
| Password + bcrypt login (`SEED_ADMIN_PASSWORD`, `passwordHash`) | **OTP-only** login via perma-OTP `666666` (OtpCode rows, expiry 2099-12-31). `User.pwdhash` exists but is not the login path. QA logs in via `DEMO_OTP=666666`. |
| `firstName` / `lastName` | Single `User.name` |
| Single `User.role` + a `CAREGIVER` role | `User.roles UserRole[]` (array). `UserRole = PATIENT, PROVIDER, MEDICAL_DIRECTOR, HEALPLACE_OPS, SUPER_ADMIN` — **no CAREGIVER** |
| `User.status: 'INACTIVE'` | `User.accountStatus AccountStatus`; `AccountStatus = ACTIVE, BLOCKED, SUSPENDED` — **no INACTIVE** (use SUSPENDED/BLOCKED) |
| — | `onboardingStatus` (NOT_COMPLETED/COMPLETED) **and** `enrollmentStatus` (NOT_ENROLLED/ENROLLED) are orthogonal gates |
| One assignment row, multi-practice per patient | `User.providerAssignmentAsPatient` is **single/unique** — a patient has exactly one `PatientProviderAssignment` (one practice) |
